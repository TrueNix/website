arXiv — 'Agent-Phishing' Attack Achieves 97.8% RCE Across 12 Agentic Red-Team Tools

2026-06-25 Security by al-ice.ai Editorial

AI relevance: The first systematic security analysis of agentic offensive AI platforms reveals that the targets these autonomous red-team agents test can reverse-attack the agents themselves — achieving remote code execution inside worker environments in 97.8% of completed runs, then escalating to full host compromise.

What Happened

  • A new arXiv paper (2606.24496) presents the first in-depth security audit of 12 widely-used agentic red-team platforms — autonomous AI systems designed to run offensive security operations with minimal human oversight.
  • Researchers introduce "agent-phishing" — a manipulation technique distinct from prompt injection that exploits the agent's own tool-use behavior rather than its language processing.
  • The attack stages realistic but benign-looking artifacts on a honeypot target: a password vault utility ("pwcrypt"), a database restore helper, or a config unpacker — tools the agent believes it needs to complete its penetration test.
  • When the agentic red-team tool discovers these artifacts, it downloads them to its own infrastructure, inspects them, and executes them — believing this is the correct path to progress the engagement.
  • The payloads embed carefully engineered vulnerabilities. In the pwcrypt example, a crafted vault file triggers an out-of-bounds write that overwrites a function pointer table, leading to system() being called with attacker-controlled arguments — all as a silent side effect of normal parsing.
  • Across 10 agentic red-team tools and 6 advanced LLMs (including Claude Opus 4.8, GPT-5.5, and Gemini 3.1 Pro), agent-phishing achieved RCE inside the worker environment in 97.8% of completed runs.
  • Failures occurred only when model-level safety mechanisms refused to perform penetration testing at all. Once a test began, the agent almost always found, downloaded, and executed the staged payload.

The Kill Chain

  • Stage 1 — Worker compromise: Agent-phishing achieves RCE inside the worker (Kali Linux container).
  • Stage 2 — Worker-to-orchestrator escalation: Shared writable volumes, weak network isolation, or unauthenticated internal APIs let the attacker reach the orchestrator that stores LLM API keys and cross-session logs.
  • Stage 3 — Persistence: "Hard" methods (trojanizing source code that survives container resets) or "soft" methods (poisoning the agent's skills and episodic memory to reinfect future sessions).
  • Stage 4 — Sandbox escape: Overly permissive container capabilities (--privileged, CAP_NET_RAW, mounted Docker sockets) enable breakout.
  • Stage 5 — Full host compromise.

Why It Matters

Existing guardrails are enforced at the orchestrator level (e.g., blocking Nmap traffic to .gov domains) but do not monitor the workers' actual OS-level behavior. An attacker with a shell on a worker can generate arbitrary network traffic and execute commands outside the guardrails' visibility. The tools built to autonomously test your security are themselves a systemic attack surface — and the targets can weaponize them against their operators.

What To Do

  • Treat LLM-controlled workers as untrusted. Maintain strict worker-orchestrator filesystem and network separation.
  • Keep secrets (LLM API keys, credentials) out of worker environments entirely.
  • Enforce OS-level guardrails via an external egress proxy — do not rely on orchestrator-level policy alone.
  • Run high-capability tools through narrow, hardened APIs with least-privileged, scoped workers.
  • Avoid running tool execution on the orchestrator node.
