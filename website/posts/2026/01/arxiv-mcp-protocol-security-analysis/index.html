arXiv — MCP protocol security analysis: capability attestation & sampling injection

2026-01-30 • Category: Research

  • What it is: a protocol-level security analysis of the Model Context Protocol (MCP), focused on how tool-integrated agents authenticate (or fail to authenticate) what servers claim and what servers ask clients to do.
  • Claimed vulnerability #1: no capability attestation — a server can claim broad permissions without a protocol-native way for the client to verify those claims.
  • Claimed vulnerability #2: bidirectional sampling without origin authentication — server-initiated “sampling” can become a prompt-injection channel back into the client’s model context.
  • Claimed vulnerability #3: implicit trust propagation in multi-server setups — a weak or malicious server can influence workflows that end up invoking other servers/tools.
  • Measurement: authors describe running hundreds of attack scenarios across multiple MCP server implementations and report higher success rates vs non-MCP equivalents (attributed to protocol design choices).
  • Proposed fix: a backward-compatible extension (“MCPSec”) with capability attestation + message authentication, with small reported latency overhead.

Why it matters

  • Protocols become defaults: once MCP is “the way tools connect,” its security properties (good or bad) get inherited by every agent and every marketplace server.
  • “Prompt injection” becomes a transport problem: when servers can solicit model completions, the attack surface isn’t just the content you read — it’s the protocol messages you accept.
  • Enterprise agent sprawl: multi-server and multi-tool graphs are exactly where trust propagation issues become real (and hard to reason about).

What to do

  1. Inventory MCP servers: treat them like dependencies with SBOM-like tracking (owner, version, permissions, data touched).
  2. Constrain permissions outside the protocol: run servers in sandboxes/containers; scope filesystem mounts; isolate credentials per server.
  3. Gate sampling/tool calls: require explicit user confirmation (or policy allowlists) for server-initiated actions that can change model context or trigger tool execution.
  4. Log and alert: record the full chain of MCP interactions (server → request → tool invocation) so you can detect cross-server “weird graphs.”
