arXiv — “Breaking the Protocol”: MCP’s protocol-level security gaps

2026-01-30 • Category: Research

  • A new paper claims the Model Context Protocol (MCP) has protocol-level design issues that increase attack leverage in tool-integrated agents, even if individual servers are well implemented.
  • Three highlighted gaps:
    • No capability attestation: a server can claim arbitrary permissions/capabilities without a strong verification mechanism.
    • Bidirectional sampling without origin authentication: server-initiated “sampling” can act as a channel for server-side prompt injection unless messages are authenticated/attributed.
    • Implicit trust propagation: in multi-server setups, trust can “flow” across components in ways that are hard to reason about and contain.
  • The paper reports an evaluation over 847 attack scenarios across multiple MCP server implementations, and argues MCP integration amplifies attack success rates versus equivalent non-MCP integrations.
  • It proposes a mitigation approach (“AttestMCP”) adding capability attestation and message authentication, with a reported low median latency overhead.
  • Even if you don’t buy every number, the central ops takeaway holds: protocol primitives shape your threat model—not just individual tool implementations.

Why it matters

  • Teams adopting MCP often focus on “is this server safe?”, but protocol-level gaps can create new classes of cross-tool and cross-server exploit chains.
  • As MCP becomes glue between IDEs, agent runtimes, and infra tools, missing provenance/attestation becomes a scalable problem (supply-chain + runtime trust).

What to do

  • Threat-model sampling: treat server-initiated LLM sampling as an injection surface; log it distinctly and gate it behind policy.
  • Adopt explicit allowlists: pin MCP servers (hash/version), and prefer curated registries over “run random npx/uvx from a blog post”.
  • Isolate servers: run MCP servers with minimal OS/network privileges; avoid sharing credentials across servers.
  • Add provenance to your agent traces: record which server/tool produced which text that entered the model context (helps post-incident forensics).
  • Design for containment: assume injection succeeds sometimes; focus on limiting the set of actions a compromised context can trigger.
