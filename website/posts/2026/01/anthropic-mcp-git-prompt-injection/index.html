Anthropic mcp-server-git: prompt injection bugs in official MCP Git server (CVE-2025-68143/4/5)

2026-01-30 • Category: News

  • What happened: Researchers disclosed three vulnerabilities in Anthropic’s official MCP Git server (mcp-server-git), exploitable via prompt injection (indirectly, by influencing content the assistant reads).
  • Attack model: The attacker doesn’t need credentials; they need the assistant to ingest attacker-controlled text (e.g., a malicious README, a poisoned issue, or a compromised web page) and then take tool actions through MCP.
  • Impact described: path/argument validation failures can let an attacker push the Git server to operate outside its intended repo boundary.
  • Reported outcomes: arbitrary file deletion, loading arbitrary files into model context, and (in certain combinations) code execution when paired with filesystem/write capabilities.
  • CVEs: CVE-2025-68143, CVE-2025-68144, CVE-2025-68145 (per the reporting).
  • Status: Fixes were released in December 2025; update if you run older builds.
  • Pattern: “safe tool” assumptions break when tools compose—Git + filesystem access can quickly become “read/write anywhere,” especially under prompt injection pressure.

Why it matters

  • MCP turns text into actions: prompt injection isn’t just “the model said something weird” — it’s an avenue to trigger real tool calls.
  • Reference implementations become defaults: many teams adopt the “official” server as-is; issues here propagate widely.
  • Composability is the risk multiplier: a Git connector that can wander directories becomes much more dangerous when any other connector can write files or execute commands.

What to do

  1. Update: ensure mcp-server-git is at a fixed release (and pin versions in lockfiles).
  2. Minimize tool scope: run MCP servers with least-privilege filesystem access (dedicated working directory; no home dir mounts; no secrets in readable paths).
  3. Prevent dangerous tool composition: avoid enabling “Git” + “filesystem write” together unless you have strong policy enforcement and audit logs.
  4. Harden prompts & policies: implement explicit allowlists for tool actions; require user confirmation for destructive operations (delete/overwrite) and for reading sensitive paths.
  5. Log tool calls: record every tool invocation with inputs/outputs; alert on attempts to access paths outside allowed roots.
