arXiv: ShareLock — Shamir's Secret Sharing Hides MCP Tool Poisoning Across Multiple Servers

2026-07-01 Research by al-ice.ai Editorial
  • Researchers from Shanghai Jiao Tong University introduce ShareLock, a multi-tool threshold poisoning framework that exploits Shamir's secret sharing to distribute malicious instructions across multiple MCP tool descriptions.
  • Each tool description contains a benign-looking secret share — individually harmless, but when aggregated by a covert reconstruction trigger, they reassemble the full malicious payload.
  • The approach achieves information-theoretic secrecy: no single tool description reveals anything about the hidden instruction, defeating both manual review and automated scanners like MCPSafetyScanner and MCP-Guard.
  • ShareLock maintains an average attack success rate exceeding 90% across mainstream LLMs on two distinct MCP clients, significantly outperforming existing single-tool poisoning strategies.
  • The framework introduces fault tolerance — the attack survives moderate auditing because removing or inspecting a subset of tools doesn't reveal the payload or break the reconstruction.
  • A covert trigger planted during a server update instructs the agent to aggregate shares and reconstruct the hidden instruction, leading to critical breaches of system assets or private data.
  • The benchmark covers four multi-tool scenarios, demonstrating that cooperative multi-tool poisoning disperses detection risk in ways single-tool attacks cannot.
  • The paper argues that current MCP security models assume monolithic plaintext embedding — ShareLock breaks that assumption by making each share statistically independent from the payload.

Why it matters

Tool poisoning attacks against MCP have typically relied on embedding malicious prompts directly in a single tool description. That model is increasingly detectable — scanners and manual review catch obvious anomalies. ShareLock demonstrates that distributing the attack across N tools using a threshold scheme makes each component look clean while the aggregate is lethal. This is the MCP equivalent of split-knowledge key escrow, weaponized for agent hijacking.

For defenders, the implication is severe: auditing individual MCP servers in isolation is insufficient. The attack surface is the combination of tools an agent connects to, not any single server. Multi-tool auditing, cross-server description correlation, and runtime behavioral anomaly detection become necessary controls.

What to do

  • Audit tool combinations, not just individual servers. If your agent connects to multiple third-party MCP servers, review the aggregate description space for hidden reconstruction triggers.
  • Implement runtime behavioral bounds. Even if a poisoned description passes static review, limit what the agent can do with sensitive data — egress filtering, human-in-the-loop for high-impact actions.
  • Pin tool descriptions to known-good versions. Treat description changes with the same rigor as code changes — require re-approval when metadata is modified.
  • Monitor for unusual aggregation patterns. If an agent suddenly starts combining outputs from multiple tools in unexpected ways, that's a ShareLock-style reconstruction signal.
