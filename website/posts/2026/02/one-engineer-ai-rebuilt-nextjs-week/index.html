The $1,100 Framework: When AI Rewrites the Rules of Software

One engineer. One AI model. Seven days. They rebuilt the most popular front-end framework from scratch—and it runs 4x faster.

$1,100

Total cost in Claude API tokens to rebuild Next.js

Let that number sink in. For the price of a decent laptop, Cloudflare's Steve Faulkner and Claude didn't just adapt Next.js or wrap it—they reimplemented the entire API surface on Vite. Two routers. 33+ module shims. Server rendering pipelines. React Server Components. File-system routing. Middleware. Caching. Static export.

The result? vinext—a drop-in replacement that builds production apps up to 4x faster and produces client bundles 57% smaller. It's already running in production for government sites.

This isn't a story about a faster framework. This is a story about what happens when AI removes the need for abstraction layers we built to manage human limitations.

◆ ◆ ◆

The Layers We Built to Think

Why do we have so many layers in the software stack? Why frameworks on top of frameworks, wrapper libraries, thousands of lines of glue code?

Because humans need help. We couldn't hold the whole system in our heads, so we built abstractions to manage complexity. Each layer made the next person's job easier. That's how you end up with OpenNext—a heroic effort to reverse-engineer Next.js build output and reshape it for platforms like Cloudflare Workers, Netlify, or AWS Lambda.

It works. But it's fragile. Every Next.js version brings unpredictable changes. It's whack-a-mole engineering.

Faulkner's insight: What if we didn't adapt the output? What if we just rebuilt the API on a better foundation?

Normally, this would take a team of engineers months, if not years. Several companies have tried. The scope is enormous.

This time it took under a week.

"AI doesn't have the same limitation. It can hold the whole system in context and just write the code. It doesn't need an intermediate framework to stay organized. It just needs a spec and a foundation to build on."
— Steve Faulkner, Cloudflare

The Numbers Don't Lie

Build Speed

Next.js 16 (Turbopack): 7.38s
vinext (Vite 7): 4.64s — 1.6x faster
vinext (Vite 8): 1.67s — 4.4x faster

Bundle Size (gzipped)

Next.js 16: 168.9 KB
vinext (Rollup): 74.0 KB
vinext (Rolldown): 72.9 KB — 57% smaller

These aren't synthetic benchmarks. This is a real 33-route App Router application. Same work. Same features. Dramatically better performance.

And here's the kicker: npm install vinext, replace next with vinext in your scripts, and everything else stays the same. Your existing app/, pages/, and next.config.js work as-is.

Why This Worked (And Why It Matters)

Not every project would go this way. This one did because several things lined up at the right moment:

Next.js is well-specified. Extensive documentation. Massive user base. Years of Stack Overflow answers. The API surface is all over the training data. When you ask Claude to implement getServerSideProps, it doesn't hallucinate.

Next.js has an elaborate test suite. Thousands of E2E tests covering every feature and edge case. Faulkner ported tests directly from their suite. This gave him a specification he could verify against mechanically.

Vite is an excellent foundation. It handles the hard parts: fast HMR, native ESM, clean plugin API, production bundling. They didn't have to build a bundler. They just had to teach it to speak Next.js.

The models caught up. Earlier models couldn't sustain coherence across a codebase this size. New models can hold the full architecture in context, reason about how modules interact, and produce correct code often enough to keep momentum going.

Take any one of these away and this doesn't work.

But here's what matters: This pattern will repeat.

The Abstraction Reckoning

Most abstractions in software exist because humans need help managing complexity. We built layers to stay organized, to make the next person's job easier, to keep the system comprehensible.

AI doesn't have that constraint. It can hold the whole system in working memory. It doesn't need intermediate frameworks to stay organized.

So which abstractions are truly foundational, and which ones were just crutches for human cognition?

That line is about to shift. Hard.

vinext is a data point. They took an API contract, a build tool, and an AI model, and the AI wrote everything in between. No intermediate framework needed. The layers we've built up over the years aren't all going to make it.

This isn't about AI replacing developers. Faulkner was steering the entire time—architecture decisions, prioritization, knowing when the AI was headed down a dead end. The human still has to conduct the orchestra.

But the orchestra just got a lot bigger. And a lot cheaper.

What You're Not Being Told

The tech press will frame this as "AI makes development faster." That's the boring take.

The real story: We're about to find out which parts of our stack were solving human problems versus actual technical problems.

How many wrapper libraries exist because humans needed simpler APIs? How many abstraction layers were built to manage cognitive load, not computational complexity? How many frameworks were just elaborate scaffolding for human limitations?

AI can work directly with the complexity. It doesn't need the scaffolding.

This is the Abstraction Reckoning. And it's going to be brutal for a lot of software that exists primarily to make other software easier to use.

"It's not clear yet which abstractions are truly foundational and which ones were just crutches for human cognition. That line is going to shift a lot over the next few years."

Your Move

If you're building software, ask yourself: Is my abstraction layer solving a technical problem, or a human problem?

If it's the former, you're probably safe. If it's the latter, you're in the blast radius.

The cost of building software just dropped by orders of magnitude. Not for everything. Not yet. But for well-specified problems with good test suites and solid foundations? The game just changed.

One engineer. Seven days. $1,100.

What are you going to build?
